## 🔍 옵션별 상세 분석

### 옵션 A: 썸네일만 추출 (Thumbnail-Only)

#### 구현 내용
```
작업:
1. ResourceListInfo JSON 파싱
2. 각 리소스의 Base64 썸네일 디코딩
3. PNG로 저장 (resource/intro/images/THUMB_*.png)
4. RESOURCE_INVENTORY.json 생성

결과:
- 185개 저해상도 썸네일 (평균 30KB)
- 메타데이터만 포함 (실제 파일 경로 없음)
```

#### 장점 ✅

```
1. 즉시 실행 가능
   → 바로 지금 실행하면 5초 만에 완료
   → 리소스 현황을 빠르게 파악 가능

2. 개발 비용 최소
   → 이미 추출된 데이터 활용
   → 새로운 바이너리 파싱 불필요

3. 저장 공간 최소
   → 전체 50MB 정도 (원본 Mars 압축 해제 불필요)
   → 저사양 시스템에서도 무리 없음

4. 빠른 시각화
   → VS Code 이미지 프리뷰로 185개 리소스 한눈에 확인
   → 썸네일로도 대부분의 리소스 식별 가능

5. 위험도 최소
   → 원본 Mars 파일 건드리지 않음
   → 실패해도 영향 없음
```

#### 단점 ❌

```
1. 편집 불가능
   → 저해상도 썸네일로는 수정 작업 불가
   → 실제 리소스를 얻으려면 나중에 다시 추출 필요

2. 메타데이터 불완전
   → 파일명, 포맷, 크기 등 일부 정보만 제공
   → 실제 파일 경로 미포함

3. 나중에 원본 추출 필요
   → 리소스 재사용하려면 결국 Step 2에서 추출
   → 이중 작업 가능성

4. Mars 재구성 불가
   → 썸네일로는 원본 파일 복원 불가
   → 수정된 리소스를 Mars에 다시 포함시킬 수 없음

5. 제한적 활용
   → 참고/검토용으로만 사용 가능
   → 실제 제작에는 부적합
```

#### 사용 시나리오

```
✅ 적합:
- "인트로.mars에 어떤 리소스가 있는지 빠르게 확인"
- "썸네일로 충분한 현황 파악"
- "시간이 급할 때 빠른 검토"

❌ 부적합:
- "리소스를 직접 수정하고 싶음"
- "기존 Mars의 리소스를 재사용하려 함"
- "완전한 에셋 라이브러리 구축"
```

#### 적용 시점

```
→ 프로젝트 초기 스커핑 단계 (시간 제약이 있을 때)
→ 신입 팀원에게 프로젝트 리소스 소개할 때
→ 빠른 프로토타입 검토
```

### 옵션 B: 모든 파일 완전히 추출 (Full Extraction)

#### 구현 내용
```
작업:
1. Resources 섹션 GZIP 해제
2. 바이너리 구조 파싱
   - 각 파일의 경로와 데이터 경계 식별
   - 파일 포맷별 메타데이터 추출
3. 각 파일을 개별 추출
4. 원본과 수정본 관리 메커니즘 구축

결과:
- 185개 원본 파일 (이미지, 모델, 오디오 등)
- 완전한 메타데이터
- 변경 추적 시스템
- ~200MB 저장 공간
```

#### 장점 ✅

```
1. 완벽한 리소스 관리
   → 모든 원본 파일에 접근 가능
   → 품질 손실 없음 (원본 그대로)

2. 무제한 편집 자유도
   → 모든 파일을 직접 수정 가능
   → Photoshop, Blender, Audacity 등 모든 도구 사용 가능

3. 완전한 추적 가능
   → 모든 변경사항 세밀하게 추적
   → 히스토리 관리 가능

4. Mars 재구성 가능
   → 수정된 파일을 원래 Mars에 다시 포함 가능
   → 완벽한 라운드트립 가능

5. 장기 에셋 라이브러리
   → 향후 모든 프로젝트에서 재사용 가능
   → 한 번의 추출로 영구 자산화

6. 최고 품질 유지
   → 원본 해상도 유지
   → 손실 없는 수정
```

#### 단점 ❌

```
1. 구현 복잡도 매우 높음
   → Resources 바이너리 구조 완벽히 파악 필요
   → 파일 경계 식별 로직 복잡
   → 예상 개발 시간: 2-3주

2. 추출 시간 오래 걸림
   → 실행 시간: 20-30분
   → 나중에 다시 실행하려면 시간 많이 소비

3. 저장 공간 많이 필요
   → 기본 200MB (이상)
   → 향후 확장 시 수GB 가능
   → 팀 협업 시 공유 저장소 부담

4. 재포장 로직도 복잡
   → 수정된 파일을 다시 Mars에 포함
   → 바이너리 구조 이해 필수
   → 테스트 여러 번 필요

5. 오류 위험 높음
   → 바이너리 파싱 오류 시 전체 실패
   → 파일 손상 가능성
   → 롤백 전략 필요

6. 유지보수 부담
   → 바이너리 구조 변경 시 스크립트 수정
   → 문서화 필수
   → 팀 지식 공유 필요

7. 불필요한 추출
   → 실제로는 10-20개 리소스만 편집
   → 나머지 170개 추출은 낭비
```

#### 사용 시나리오

```
✅ 적합:
- "모든 기존 리소스를 새 프로젝트에서 재사용"
- "완벽한 에셋 라이브러리 구축이 목표"
- "장기적으로 여러 Mars 파일 통합"
- "개발 시간이 충분함"

❌ 부적합:
- "시간이 급함 (1주일 내 결과 필요)"
- "소수의 리소스만 편집할 것 같음"
- "팀의 PowerShell 역량이 낮음"
- "저사양 시스템에서 작업"
```

#### 적용 시점

```
→ 장기 프로젝트 (6개월 이상)
→ 여러 Mars 파일을 통합할 때
→ 에셋 라이브러리 구축이 우선순위
→ 전담 개발자가 있을 때
```

### 옵션 C: 하이브리드 (Hybrid) ⭐ **추천**

#### 구현 내용
```
단계 1 (즉시):
- 썸네일 추출 (완료 ✅)
- RESOURCE_INVENTORY.json 생성
- 현황 파악

단계 2 (선택적, 차주):
- 필요한 리소스만 추출 (5-10개)
- 원본 파일 워크스페이스에 배치
- 편집 가능 상태로 전환

단계 3 (필요시):
- 변경사항 추적
- Mars 재구성
- 최종 통합

결과:
- 빠른 초기 확인 (1초)
- 유연한 확장 (필요한 만큼만)
- 완벽한 품질 유지 (필요한 것은)
- 효율적 저장 공간 (50-150MB)
```

#### 장점 ✅

```
1. 즉시 시작 가능
   → Step 1 지금 바로 실행 (5초)
   → 빠른 결과 → 빠른 피드백

2. 점진적 확장
   → 필요에 따라 단계적으로 진행
   → 중간에 우선순위 변경 가능

3. 효율적 개발
   → 필요한 것만 구현 → 개발 시간 단축
   → 단계별 검증 → 오류 조기 발견

4. 저장 공간 최적화
   → 초기: 50MB (썸네일)
   → 중간: 100-150MB (선택 리소스)
   → 완전: 200MB (필요 시에만)

5. 유연한 의사결정
   → Step 1 결과로 Step 2 여부 결정
   → 리소스 선택 후 추출 여부 결정
   → 비용/이득 실시간 판단

6. 위험 분산
   → 각 단계별로 오류 처리 가능
   → 한 단계 실패해도 이전 단계는 유지
   → 롤백 간편

7. 팀 협업 용이
   → 단계별 명확한 역할 분담
   → 병렬 작업 가능 (Step 1과 MCP 동시 진행)
   → 문서화 단순

8. 추후 마이그레이션 용이
   → 언제든지 Option B로 전환 가능
   → 완전 자동화도 가능
   → 최적화 경로 자유로움
```

#### 단점 ❌

```
1. 초기 계획 필요
   → 단계별 목표 명확화 필요
   → 리소스 선택 기준 정의
   → ~1-2시간 계획 투자

2. 복수 스크립트 관리
   → 3-4개의 별도 스크립트 필요
   → 파라미터 관리 복잡도 중간
   → 문서화 여러 개

3. 상태 추적 메커니즘
   → .sha256, .status.json 등 추가 파일
   → 변경 감지 로직 필요
   → 초기 설정 시간

4. 최종 통합까지 시간 소비
   → 각 단계마다 1-2주
   → 단계 3까지 4-6주 예상
   → 전체 구현 기간 길어짐 (Option D 대비)
```

#### 사용 시나리오

```
✅ 적합 (= 현재 프로젝트):
- "빠른 초기 확인이 필요" → Step 1 (지금)
- "향후 일부 리소스 수정할 것 같음" → Step 2 (차주)
- "기존과 신규 콘텐츠를 혼합" → Step 3 (필요시)
- "개발 기간과 품질의 균형"
- "팀의 제한된 리소스로 최대 효과"

❌ 부적합:
- "모든 리소스를 지금 당장 편집해야 함"
- "단일 큰 프로젝트로 완성해야 함" (Option B가 나음)
```

#### 적용 시점

```
→ 중소 프로젝트 (2-3개월)
→ 우선순위가 명확할 때
→ 단계적 진행이 가능할 때
→ 현재 프로젝트 상황에 최적 ⭐
```

### 옵션 D: 신규 콘텐츠 중심 (New Content Focus)

#### 구현 내용
```
전략:
- 기존 Mars는 참고용만 (읽기)
- 모든 새 콘텐츠는 MCP + MarsCatalogue로 생성
- 기존 리소스 추출 불필요
- 점진적 마이그레이션

결과:
- 깨끗한 MarsCatalogue 구조
- 새로운 표준에 따른 일관성
- 기존과의 혼재 없음
```

#### 장점 ✅

```
1. 가장 빠른 시작
   → 추출 작업 전혀 불필요
   → 바로 MCP로 콘텐츠 생성 시작
   → 개발 시간: Option A 수준

2. 깔끔한 구조
   → 기존 리소스의 복잡성 없음
   → MarsCatalogue 표준 준수
   → 유지보수 간단

3. 미래 지향적
   → 최신 도구와 표준 사용
   → 기술 부채 없음
   → 확장성 좋음

4. 명확한 구분
   → 기존 Mars와 신규 완전히 분리
   → 혼동 없음
   → 버전 관리 단순

5. 병렬 진행 최적
   → 리소스 추출과 무관하게 진행
   → 팀 역할 분담 명확
   → 개발 속도 빨라짐

6. 최소 저장 공간
   → 필요한 새 리소스만 저장
   → 기존 추출 불필요
   → 저사양 환경 친화

7. 위험도 최소
   → 기존 파일 건드리지 않음
   → 실패 영향 최소
   → 롤백 불필요
```

#### 단점 ❌

```
1. 기존 리소스 재사용 불가
   → 169개 이미지, 14개 모델 모두 무시
   → 새로 생성해야 함 (또는 수동 복사)

2. 초기 리소스 준비 필요
   → 이미지, 모델 등을 직접 준비해야 함
   → 디자인 작업 필요
   → 리소스 확보 시간 증가

3. 데이터 이전 비용
   → 나중에 기존 Mars와 통합하려면 수동 작업
   → 점진적 마이그레이션 필요
   → 추가 업무

4. 기존 콘텐츠 활용 불가
   → "인트로.mars"의 텍스트, 배치 등 참고 불가
   → 스크래치부터 시작
   → 개발 시간 증가 가능

5. 단기 결과물 부족
   → 신규 리소스 확보 전까지 콘텐츠 제한
   → 초기 데모용 콘텐츠 제약

6. 기존 투자 낭비
   → "인트로.mars"의 185개 리소스를 활용하지 않음
   → 제작 비용 낭비 느낌
```

#### 사용 시나리오

```
✅ 적합:
- "기존 Mars는 완전히 버리고 새로 시작"
- "새로운 디자인/구조로 완전 리뉴얼"
- "신규 프로젝트로 기존과 독립" (= 다른 프로젝트)
- "충분한 리소스로 처음부터 제작"

❌ 부족:
- "기존 리소스를 재사용하고 싶음" (= 현재 프로젝트)
- "시간과 비용 절감이 목표"
- "기존 콘텐츠의 기반 유지"
```

#### 적용 시점

```
→ 신규 독립 프로젝트
→ 기존과의 호환성 불필요
→ 리소스 개발 팀이 별도 있을 때
→ 중장기 프로젝트로 충분한 시간 있을 때
```
